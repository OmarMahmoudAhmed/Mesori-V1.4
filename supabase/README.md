# قاعدة بيانات Mesori (Supabase)

## كيفية التطبيق
1. افتح مشروعك على [Supabase Dashboard](https://app.supabase.com) → **SQL Editor**.
2. انسخ محتوى `migrations/001_schema.sql` بالكامل والصقه، ثم اضغط **Run**.
   - الملف يبدأ بـ `DROP TABLE IF EXISTS ... CASCADE` لكل الجداول، فتشغيله آمن
     حتى لو كانت الجداول موجودة بالفعل من قبل (مثل السكيما القديمة المكسورة) —
     هيمسحها ويعيد إنشاءها بالشكل الصحيح من الصفر.
3. بعد نجاح الاستعلام، شغّل `node scripts/seed.js` من جذر المشروع لرفع
   المستويات والمراحل والأسئلة (يحتاج `.env` فيه `VITE_SUPABASE_URL` و
   `SUPABASE_SERVICE_ROLE_KEY` — راجع `.env.example`).

## لماذا `DROP TABLE` وليس `ALTER TABLE`؟
السكيما القديمة كانت تحتوي على مشكلة بنيوية (id غير فريد عبر المستويات/المراحل)
تسببت في فقدان بيانات صامت أثناء الرفع الأول (نجت فقط بيانات آخر مستوى/مرحلة
تمت معالجتها). بما أن المشروع لم يُطلق بعد ولا يوجد مستخدمون حقيقيون، فحذف
الجداول القديمة (التي أصلاً بياناتها ناقصة) وإعادة الرفع بالكامل أبسط وأضمن
من محاولة ترقيع بيانات جزئية تالفة.

## هيكل المفاتيح
- `stages`: مفتاح أساسي مركّب `(level_id, id)` — لأن `id` يتكرر 1-5 داخل كل مستوى.
- `questions`: مفتاح أساسي مركّب `(level_id, stage_id, id)` — لأن `id` يتكرر 1-10
  داخل كل مرحلة. هذا التصميم متعمّد في `src/data/*.js` (id محلي وليس عام)،
  والسكيما الآن تطابقه بدل افتراض تفرّد `id` عالمياً.

## ملاحظة عن profiles / user_progress
الجدولان جاهزان بسياسات RLS صحيحة (كل مستخدم يرى/يعدّل صفّه فقط)، لكن التطبيق
الحالي **لا يستخدمهما بعد** لأنه لا يوجد تسجيل دخول حقيقي (auth.uid() فاضية
دايماً بدونه). التقدم والنقاط لسه بتتحسب محلياً في AppContext.jsx كما كانت،
لحد ما تُضاف صفحة تسجيل الدخول (راجع BACKEND_GUIDE.md — المرحلة 3).
